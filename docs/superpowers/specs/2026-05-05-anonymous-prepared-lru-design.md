# Spec: Client-side prepared statement cache split (Named + Anonymous LRU)

Author: pg_doorman team
Date: 2026-05-05
Status: design

## Problem

Per-client prepared statement cache (`PreparedStatementCache` in
`src/client/core.rs:38-103`) today holds both Named and Anonymous
entries in one structure: either an unlimited `AHashMap` (when
`client_prepared_statements_cache_size = 0`, the default) or a
`LruCache` of fixed size.

Two problems with the current model:

1. The Limited mode is unsafe for Named statements. When LRU evicts
   a Named entry, no `Close` is sent to the backend
   (`src/client/core.rs:57-66`). The next `Bind` issued by the client
   under the same name fails with `prepared statement does not
   exist`. This is why the default is `0` (unlimited): the only way
   to avoid breakage is to never evict.
2. There is no protection against runaway Anonymous prepared
   statements. A client that keeps generating distinct one-shot
   parameterised queries grows the per-client cache without bound,
   and tens of thousands of long-lived connections multiply this
   into multi-gigabyte memory pressure.

Anonymous and Named have different ownership semantics:

- **Named** is owned by the client. Lifetime is bounded by `Close`
  (extended) / `DEALLOCATE` (simple) / disconnect. Eviction by the
  pooler is unsafe.
- **Anonymous** is generated by the pooler under a synthetic
  `DOORMAN_<N>` name (see `src/messages/extended.rs:147-152`).
  Lifetime in the client cache is purely for plan-cache reuse; the
  client never asks for it back by name. Eviction is safe.

## Goal

Split the per-client cache into two parts and apply different
policies:

- **Named cache**: unbounded, never evicted by the pooler.
- **Anonymous cache**: LRU bounded by a new config knob, default
  `256`. Eviction drops the local `Arc<Parse>` and does nothing else;
  the underlying `DOORMAN_<N>` on the backend is left alone and
  recycled by server-level LRU
  (`src/server/server_backend.rs:124`) or `server_lifetime`
  (default 20 min, `src/config/general.rs:252-254`).

This fixes the Named-eviction safety bug by construction and gives a
sane default upper bound on Anonymous memory growth.

## Non-goals

- Pool-level cache (`src/server/prepared_statement_cache.rs`) stays
  as is.
- Server-level per-backend cache stays as is.
- No active `Close` / `DEALLOCATE` is sent on Anonymous LRU
  eviction. The cross-client sharing of `DOORMAN_<N>` (other clients
  may still hold the same hash via pool cache) makes broadcast Close
  unsafe; lazy retirement via server LRU + `server_lifetime` is
  sufficient.
- The migration (binary upgrade) blob format does not change. The
  blob already discriminates `Named` vs `Anonymous` per entry; only
  reconstruction routes into the right map.

## Design

### `PreparedStatementCache` shape

Replace the `enum` in `src/client/core.rs:38-103` with a `struct`
holding two independent maps:

```rust
pub struct PreparedStatementCache {
    named: AHashMap<String, CachedStatement>,
    anonymous: AnonymousCache,
}

enum AnonymousCache {
    Unlimited(AHashMap<u64, CachedStatement>),
    Limited(LruCache<u64, CachedStatement>),
}
```

The `Anonymous` variant continues to be selected by config: `0` →
`Unlimited`, `>0` → `Limited`. The default flips to `256` (see
configuration section).

### Routing by `PreparedStatementKey`

Existing API (`get`, `put`, `pop`, `clear`, `len`, `iter`) keeps the
same signatures. Internal routing:

| Operation | `Named(s)` | `Anonymous(h)` |
|---|---|---|
| `put` | `named.insert(s, v)` | `anonymous.put(h, v)` |
| `get` | `named.get(s)` | `anonymous.get(&h)` (bumps LRU) |
| `pop` | `named.remove(s)` | `anonymous.pop(&h)` |
| `clear` | clears both maps | clears both maps |
| `len` | sum of both | sum of both |
| `iter` | chained iterator | chained iterator |

`AnonymousCache::put` may return an evicted entry. The return value
is dropped: only the local `Arc<Parse>` is released. No backend
traffic.

### `CachedStatement` and `Arc<Parse>` ownership

Unchanged. Each entry holds `Arc<Parse>` shared with the pool cache.
When an Anonymous entry is evicted, its `Arc` count decrements; if
the pool cache has already evicted the same hash, this is the last
ref and memory is freed. Otherwise the pool cache retains
ownership.

### Cross-client `DOORMAN_<N>` lifecycle

For an Anonymous entry that is evicted by client A's LRU but still
held by client B's Anonymous cache (via the same hash and shared
`Arc<Parse>`):

- Pool cache entry: untouched.
- Client B's cache: untouched. Its next `Bind` continues to work via
  `ensure_prepared_statement_is_on_server`
  (`src/client/protocol.rs:23-89`).
- Backends that already hold `DOORMAN_<N>` in their server-level
  cache: keep it. They retire it via their own LRU
  (`src/server/server_backend.rs:124`) when newer entries push it
  out.
- Worst case lifetime of an unused `DOORMAN_<N>` on a backend:
  `server_lifetime` (default 20 min). When the backend is rotated,
  every prepared statement on it is gone.

### `pool_cache::CacheEntry`: `seen_as_*` flags

Add two atomic flags to `CacheEntry`
(`src/server/prepared_statement_cache.rs:31-51`):

```rust
seen_as_named:     AtomicBool,
seen_as_anonymous: AtomicBool,
```

`PreparedStatementCache::get_or_insert` already receives the
incoming `Parse`. We extend the call site in
`src/client/protocol.rs:128-315` (or thread it through
`register_parse_to_cache`) so that the cache learns whether the
client used a non-empty name. On every registration the appropriate
flag is set:

- non-empty client name → `seen_as_named = true`
- empty client name → `seen_as_anonymous = true`

Both flags can be true on the same entry (different clients hit the
same hash with different ownership semantics). Neither flag goes
back to `false`.

### `kind` column in `SHOW PREPARED_STATEMENTS`

Existing output: `pool, hash, name, query, count_used`. Add `kind`
column derived from the flags:

| `seen_as_named` | `seen_as_anonymous` | `kind` |
|:---:|:---:|---|
| true | false | `named` |
| false | true | `anonymous` |
| true | true | `mixed` |

The combination "both false" is unreachable: an entry only exists if
at least one client registered it.

### `SHOW POOLS_MEMORY` columns

Existing columns include `client_prepared_count` and
`client_prepared_bytes`. Add two new columns:

- `client_named_count`
- `client_anonymous_count`

`client_prepared_count` stays as the sum (backward compatible).
`client_prepared_bytes` stays as the sum.

### Prometheus metrics

Existing:

- `pg_doorman_clients_prepared_cache_entries{user, database}`
- `pg_doorman_clients_prepared_cache_bytes`

Stay as sums of both parts.

New:

- `pg_doorman_clients_prepared_named_entries{user, database}`
- `pg_doorman_clients_prepared_anonymous_entries{user, database}`
- `pg_doorman_clients_prepared_anonymous_evictions_total{user, database}`
  (counter, increments on every Anonymous LRU eviction)

The eviction counter is the primary signal for tuning
`client_anonymous_prepared_cache_size`: a sustained non-zero rate
means the limit is too small for the workload.

### Configuration

New parameter in `src/config/general.rs`:

| Field | Type | Default | Effect |
|---|---|---|---|
| `client_anonymous_prepared_cache_size` | `usize` | `256` | Per-client Anonymous LRU size. `0` → unlimited. |

`PreparedStatementState::new` takes the Anonymous limit as its sole
sizing argument. Named is always unlimited.

The old `client_prepared_statements_cache_size` is **removed**. The
`General` struct does not use `deny_unknown_fields`, so a config
that still mentions the old field parses fine and the field is
silently ignored. Operators who tuned this value lose the tuning
silently; the changelog migration note is the only signal. The new
default of `256` (vs the old `0` = unlimited) provides bounded
memory for everyone who didn't set the old value, which covers most
deployments.

### Migration (binary upgrade)

The blob format and the reconstruction logic are unchanged. The
single visible edit is a compile-required adjustment to the
`register_parse_to_cache` call site
(`src/client/migration.rs:409-418`) so that it passes the
`client_given_name` carried in the blob: the `Named(name)` case
forwards the name, the `Anonymous(hash)` case forwards `""`. This
threads the same `seen_as_*` classification path through the
migration as for live traffic. No new migration logic, no blob
versioning, no behavioural difference for clients across the
upgrade.

### `cache_memory_usage` (`src/client/core.rs:273-299`)

Iterate both maps. The `Arc::strong_count == 1` exclusion still
applies to the per-entry size of `Parse`. The shared text remains in
the pool cache and is counted there.

### Edge cases

- **Hash collision Named vs Anonymous in the same client.**
  Possible: a client may `Parse "stmt_42"` and also `Parse ""` for
  identical SQL. They land in separate maps under different keys
  (`Named("stmt_42")` and `Anonymous(hash)`). Both share the same
  `Arc<Parse>` from pool cache (with the same `DOORMAN_<N>`
  internal name). Both can be Bound independently. No interaction.

- **`DEALLOCATE name` (simple query) for an Anonymous entry.**
  Anonymous entries don't have a client-visible name. The simple
  `DEALLOCATE foo` parser only matches `Named("foo")`; the Anonymous
  map is untouched. Same as today.

- **`DEALLOCATE ALL` (simple query).** Calls `cache.clear()`, which
  now clears both maps. Same effect as today.

- **`Close` (extended F-message) with a non-empty name.** Pops from
  Named map. Anonymous map untouched. Same as today.

- **`Close` with an empty name.** No-op for client cache (today and
  in the new design). The underlying `DOORMAN_<N>` is left in pool
  cache and on backends.

- **Async clients (`DOORMAN_async_<N>`).** Async clients use unique
  per-session names (`src/client/protocol.rs:175-182`). The async
  name lives in `CachedStatement.async_name`. Async-client Anonymous
  entries land in the Anonymous map keyed by hash, exactly the same
  as sync-client Anonymous. The existing async name allocation
  remains unchanged.

- **Client disconnect mid-transaction.** Existing
  `Server::checkin_cleanup` (`src/server/server_backend.rs:383-488`)
  handles dirty server state via `DEALLOCATE ALL` when needed. Not
  affected by this change.

- **Limited mode with a `0`-sized hint.** If a downstream call ever
  passes `0` to `LruCache::new` it would panic on `NonZeroUsize`;
  the existing branch in `src/client/core.rs:236-243` covers that.
  Same logic.

## Code change layout

Commits, in order, each landing as one PR or as one commit in a
single PR:

1. **Split `PreparedStatementCache` into Named + Anonymous maps.**
   `src/client/core.rs`. Pure refactor: API signatures unchanged,
   routing by key kind. Plus unit tests on the new structure.
2. **Add `client_anonymous_prepared_cache_size` config and remove
   the old field.** `src/config/general.rs`,
   `PreparedStatementState::new` call sites. Default `256`.
3. **Add `seen_as_named` / `seen_as_anonymous` flags to
   `CacheEntry`.** `src/server/prepared_statement_cache.rs`,
   `register_parse_to_cache` signature now carries
   `client_given_name`. Update call sites in
   `src/client/protocol.rs` and `src/client/migration.rs`.
4. **`SHOW PREPARED_STATEMENTS`: `kind` column.** `src/admin/*`.
5. **`SHOW POOLS_MEMORY`: `client_named_count`,
   `client_anonymous_count`.** `src/admin/*`.
6. **Prometheus metrics.** `src/prometheus/*`. New gauges and
   eviction counter.
7. **BDD scenarios.** `tests/bdd/features/`. Cover Named survival
   under tight Anonymous limit, Anonymous eviction not breaking
   other clients sharing the same hash, `kind` classification,
   `SHOW POOLS_MEMORY` columns.
8. **Documentation.** Update
   `documentation/{en,ru}/src/tutorials/prepared-statements.md`
   "Cache layers" and "Configuration" sections;
   `documentation/{en,ru}/src/reference/general.md` for the new
   field; `changelog.md` migration note about the removed field.

## Test plan

Unit tests:

- `PreparedStatementCache`: Named put/get/pop/clear; Named never
  evicted under any Anonymous-limit value; Anonymous put with limit
  N evicts oldest after N+1 inserts; Anonymous put returns evicted
  entry; combined `iter` yields all entries; `len` returns the sum.
- `CacheEntry::seen_as_*`: starts `false`/`false`; non-empty name
  flips Named flag; empty name flips Anonymous flag; both can be
  true after mixed usage.
- Config parser: new field default is `256`; absence of old field is
  fine; presence of removed old field handled per the chosen serde
  policy.

BDD scenarios:

- Client A creates 300 distinct anonymous prepared (limit 256), then
  rebinds the oldest one. Expect: re-`Parse` happens transparently,
  no client error.
- Client A creates a Named "stmt_one", then 1000 distinct Anonymous.
  Bind "stmt_one" still succeeds without re-Parse from client side
  (Named was not evicted).
- Two clients hit the same anonymous query; client A evicts; client
  B's Bind on the shared hash continues to work.
- `SHOW PREPARED_STATEMENTS` after mixed traffic shows correct
  `kind` per entry.

Manual check:

- Long-running soak with a synthetic generator producing 10k unique
  anonymous queries. Verify
  `pg_doorman_clients_prepared_anonymous_evictions_total` grows,
  per-client memory plateaus, no errors observed by clients.

## Risks and mitigations

- **Backend memory creep.** Lazy eviction lets `DOORMAN_<N>` linger
  on backends until server-level LRU or `server_lifetime` retires
  them. Bounded by `prepared_statements_cache_size` (default 8192)
  per backend and 20 min worst-case backend rotation. Acceptable.
- **Default 256 too small for some workloads.** New eviction
  counter exposes this. Operators tune the value upward if needed.
  Existing deployments that relied on unlimited (default 0 in old
  config) get a smaller cache by default; this is the deliberate
  trade-off — bounded memory over unbounded.
- **Old config field removal breaks some users.** Migration note in
  changelog. Recommend setting
  `client_anonymous_prepared_cache_size` explicitly during the
  upgrade.
