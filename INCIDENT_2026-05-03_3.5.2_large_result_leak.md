# Инцидент 2026-05-03: pg_doorman 3.5.2 — утечка серверных коннектов на крупных результатах

## TL;DR

Утечка коннектов в pg_doorman 3.5.2. Триггер — приложение читает строку с `data` ≈ 900 МБ из таблицы `bucket_77.sku_merges WHERE sku=1841838257` (один конкретный sku). PG отдаёт ответ полностью, doorman НЕ дочитывает ответ или иначе теряет учёт коннекта — **permit утекает**, серверный backend остаётся "числиться" в pool'е. Утечка накопительная: за часы/дни пул засирается до 100% залипших. На стороне PG все 40 backend в `state=idle`, последний выполненный query — этот самый `SELECT`. Уникально для этой инсталляции (в 40 000 других нет таких больших результатов → нет утечки).

**Это не "тормоза postgresql". PG отдал ответ корректно. Утечка — bug в pg_doorman.**

## Распакованная семантика строки ошибки

Источник: `src/pool/inner.rs:1018` (v3.5.2):
```rust

    "[{}@{}] checkout timeout at phase=semaphore elapsed={}ms size={} max={} waiters={} semaphore_avail={}",
    pool_name, username, start.elapsed().as_millis(),
    slots.size, slots.max_size,
    slots.waiters.len(),                       // direct-handoff waiters в Slots, НЕ очередь Semaphore

```

- `size=42 max=40` — `slots.size` превышает `max_size` на 2.
- `waiters=0` — direct-handoff waiters внутри пула пусто (это нормально для phase=semaphore).
- `semaphore_avail=0` — ВСЕ 40 permit на руках, никто не возвращает.

## Где конкретно может утекать (код-кандидаты, v3.5.2)

### Главный подозреваемый: `handle_large_data_row` (src/server/protocol_io.rs:133)

Активируется в src/server/protocol_io.rs:498–516, когда DataRow длиннее `server.max_message_size`:

```rust
if server.max_message_size > 0
    && message_len > server.max_message_size
    && code_u8 as char == 'D'
{
    if !server.buffer.is_empty() {
        server.pending_large_message = Some((code_u8, message_len));
        server.data_available = true;
        let result = server.buffer.clone();
        server.buffer.clear();
        ...
        return Ok(result);
    }
    return handle_large_data_row(server, &mut client_stream, code_u8, message_len).await;
}
```

Тело `handle_large_data_row`:

```rust
async fn handle_large_data_row<C>(...) -> Result<BytesMut, Error> {
    server.buffer.put_u8(code_u8);
    server.buffer.put_i32(message_len);
    let prev_bad = server.bad;
    server.bad = true;
    write_all_flush(client_stream, &server.buffer).await?;       // (A) await write to client

    match proxy_copy_data_with_timeout(
        get_config().general.proxy_copy_data_timeout.as_std(),    // 15s
        &mut server.stream,
        client_stream,
        message_len as usize - mem::size_of::<i32>(),
    ).await {                                                     // (B) bounded proxy
        Ok(_) => (),
        Err(err) => {
            server.mark_bad(err.to_string().as_str());
            return Err(err);
        }
    }

    if !prev_bad {
        server.bad = false;
    }
    server.stats.data_received(server.buffer.len() + message_len as usize);
    server.last_activity = SystemTime::now();
    server.data_available = true;                                 // (C) безусловно
    server.buffer.clear();
    server.stats.wait_idle();
    Ok(server.buffer.clone())
}
```

Проблемные точки:
- **(A) `write_all_flush(client_stream)`** — без timeout. Если клиент не читает (slow / mid-disconnect), `await` зависает; снаружи нет защиты.
- **(B) `proxy_copy_data_with_timeout(15s)`** — общий 15-секундный бюджет на ВСЁ копирование, не на каждый chunk. Если ответ ~900 МБ а пропускная способность к клиенту медленнее ~60 МБ/с — таймаут наступает, `Err(ProxyTimeout)` → `mark_bad` → `return Err`. **При этом из server-сокета прочитано только то, что успели проксировать**, остальные сотни МБ остаются в kernel-буфере unix-сокета между PG и doorman. Это в точности то, что мы видим в логе через шаг ("draining unread data ... received=909M").
- **(C) `server.data_available = true` ставится безусловно** в Ok-ветке. Если PG больше ничего не пришлёт (например, ответ был только из этой одной DataRow + некий race), следующий `read_message_header` зависнет.

### Второстепенные подозреваемые

- `transaction.rs:1182–1200`: при `write_all_flush(client) Err`, `mark_bad` ставится **только** если `server.is_async() || server.in_copy_mode()`. В обычном transaction-mode: `wait_available` + `continue` loop. Это не utech permit'a, но усугубляет: продолжаем читать с сервера в "никуда".
- `wait_for_next_message` (`transaction.rs:238`) — `tokio::select!` без timeout. Полагается на TCP keepalive (1+5×5 = 26s). Keepalive работает по чтению, на pending writes не помогает; `tcp_user_timeout` дефолт 60s.
- `pre_replace_one` (inner.rs:347) делает `slots.size += 1` без `add_permits`, под условием "old сервер потом fail recycle". Но `under_pressure → skip_lifetime` (см. ниже) этот инвариант разрушает.

### Вспомогательный фактор: `server_lifetime` отключён под нагрузкой

- `retain_pool_connections` (src/pool/retain.rs:23): `if self.database.under_pressure() { return 0; }`. `under_pressure() = semaphore.available_permits() == 0`.
- `try_recycle_one` (src/pool/inner.rs:488): `let skip_lifetime = self.under_pressure();`.

Эмпирика: за 22 минуты (17:17 → 17:39) для пула `product-merge-front-shard-user` создано **6** новых backend, закрыто по lifetime — **2**. При server_lifetime=300s ожидание ~176. Это объясняет: backend живут сутками; пул содержит много "старых" коннектов.

## ROOT CAUSE (главная гипотеза, обоснована числом 42 = 40 + max_parallel_creates default 2)

### Числовое совпадение

`size=42 max=40 waiters=0 semaphore_avail=0` весь час инцидента.

В коде есть `scaling_max_parallel_creates` с **default = 2** (`src/config/general.rs:345`):
```rust
pub fn default_scaling_max_parallel_creates() -> u32 { 2 }
```

`max_size + max_parallel_creates = 40 + 2 = 42`. И ещё есть `MAX_CONCURRENT_PRE_REPLACEMENTS = 3` (`src/pool/inner.rs:240`) — pre_replace может временно растить slots.size до `max_size + 3 = 43`.

В моменте инцидента наблюдалось `42` — это **2 завершённых pre_replace_one**, каждый из которых сделал `slots.size += 1` (`src/pool/inner.rs:417`) без `add_permits` (объясняется в комментарии "No semaphore.add_permits needed").

### Почему semaphore=0 при том что 2 свежих сервера лежат в idle

- `slots.size = 42` — pool на бумаге держит 42 слота.
- `slots.vec` содержит **2 новых сервера** от pre_replacements (идут через `push_idle`).
- Остальные **40 серверов "checked out"** — у них на руках 40 permit'ов pool semaphore.
- Чтобы взять одного из 2 idle серверов, нужен permit → `acquire_semaphore()` → permits=0 → ждать → 5s timeout.
- Запросы умирают на **phase=semaphore**, не успевая дойти до try_recycle_one.

### Где сидят 40 task'ов с захваченными permit'ами

Все основные пути в client task'е либо защищены timeout'ом, либо завершаются по EOF. **Единственные места `await` без timeout, где task реально может висеть на минуты-часы при условии "TCP kernel alive, application dead"**:

| Локация | Что ждёт | Timeout |
|---|---|---|
| `src/server/protocol_io.rs:147` `write_all_flush(client_stream, &server.buffer).await?` внутри `handle_large_data_row` | flush header + buffer к клиенту перед `proxy_copy_data` | **нет** |
| `src/client/transaction.rs:1182` `write_all_flush(&mut self.write, &response).await` (для не-async / не-copy сервера) | flush ответа клиенту, при `Err` мы НЕ mark_bad и НЕ return — продолжаем loop | **нет** |
| `src/client/transaction.rs:238` `wait_for_next_message` (`tokio::select!` без timeout) | следующее сообщение от клиента в transaction-mode между запросами | полагается на TCP keepalive 26s |

`tcp_user_timeout` default=60s. Это убило бы зависший write — **но только если** kernel ACK перестал приходить от клиента. Если клиент-приложение мертво, а kernel/прокси/sidecar отвечает на keepalive ACK'и — `tcp_user_timeout` НЕ сработает, и write повиснет навечно.

### Почему 40 backend в PG в state=idle (а не FATAL)

- pg_doorman прочитал ВЕСЬ ответ PG (DataRow + CommandComplete + ReadyForQuery) — поскольку unix-socket к PG быстрый.
- backend PG ушёл в state=idle (после ReadyForQuery).
- pg_doorman завис на write к client → НЕ дошёл до Drop Object → не вернул backend в pool → не послал Terminate.
- Со стороны PG backend жив, в idle, в pg_stat_activity показывает last query = тот самый SELECT.

### Почему reserve_pool_size=20 не помог

Архитектурно: `acquire_coordinator_jit` (где выдаётся reserve) вызывается **после** успешного `acquire_semaphore`. Если pool semaphore (40) исчерпан — новые task'и таймаутятся в `phase=semaphore` (5s), до coordinator не доходят. Reserve permits координатора простаивают, потому что в архитектуре pg_doorman они **не для случая "permits перманентно залипли"**, а для "burst pressure" когда permits быстро возвращаются.

### Что показал прямой repro (BDD `@large-datarow-leak-zombie-noread*`)

Запустил тест на master и v3.5.2:
- 5 клиентов открывают сессии, отправляют `SELECT repeat('X', 50000000)` через `we send SimpleQuery without waiting` и **никогда не читают ответ**.
- Через 2 секунды `SHOW POOLS` показывает `sv_active=5` (весь pool забит, как и в проде).
- После того как тест закрывает все сессии (`we close session`), через 3 секунды `sv_active=0` — cleanup отрабатывает мгновенно.

**Это подтверждает механику**, но **не объясняет 4 дня залипания**. Если бы был **один** zombie, `tcp_user_timeout=60s` убил бы соединение со стороны kernel и pg_doorman освободил бы permit за минуту.

### Реальный сценарий 4-дневного залипания: ротация zombie

40 backend в state idle с одним и тем же query — это не **один и тот же** task hung на 4 дня, а **стабильное запруживание**:
1. Slow/zombie клиент A держит permit ≈ 60 секунд (до `tcp_user_timeout`).
2. `tcp_user_timeout` → kernel close → write fail → cleanup → permit released.
3. **Через миллисекунды** приходит другой slow клиент B (приложение в цикле retry'ит большой SELECT) — захватывает permit.
4. Цикл повторяется 4 дня.

С точки зрения снапшота `pg_stat_activity`: всегда 40 backend с тем же last_query — потому что все они **сейчас** обрабатывают тот же запрос (либо PG только что отдал ответ и ждёт next).

**Корневая проблема — не утечка, а отсутствие защиты от continuous slow-client pressure**:
- Pool semaphore (40) полностью забит slow clients навсегда.
- Reserve permits (20) бесполезны — `acquire_coordinator_jit` идёт после `acquire_semaphore`, до coordinator не доходим.
- Здоровые клиенты падают с `checkout timeout phase=semaphore` через `query_wait_timeout=5s`.
- Единственный механизм, освобождающий permit, — kernel `tcp_user_timeout` (60s default). Но при достаточном arrival rate slow clients pool никогда не "дышит".

### Что подтверждает гипотезу и что — нет

**Подтверждает:**
- `size=42` = `max_size + 2` = `40 + max_parallel_creates_default(2)`. Точное совпадение чисел.
- `semaphore_avail=0` стабильно весь час → 40 permit'ов не возвращаются → 40 task'ов в await.
- 40 backend в PG state=idle с last_query=SELECT → pg_doorman прочитал ответ полностью, но не завершил task.
- `wait_ms p99=4.96ms` ещё ДО инцидента (в 17:31) — значит несколько permit'ов уже залипли заранее (накопительно).
- `LOG: could not send data to client: Broken pipe` в PG-логе — встречается для тех pid, где **PG не уложилось** в pg_doorman recv (только 5 видимых эпизодов). Для 40 "залипших" backend такого FATAL **нет** — т.е. для них pg_doorman дочитал.

**НЕ подтверждает (или требует проверки):**
- Прямой repro `handle_large_data_row + RST + cancel` на BDD в `tests/bdd/features/large-datarow-leak.feature` (5 сценариев, включая stress 100 эпизодов и coordinator-managed pool с 2 users) — все **зелёные** на master. RST триггерит kernel-side EOF немедленно, и `write_all_flush` сразу возвращает Err → cleanup отрабатывает.
- Чтобы воспроизвести leak, нужен клиент с **живым TCP kernel + не читающей application** (медленный сервер, висящий sidecar, прокси с keepalive). Такой сценарий в чистом BDD-харнессе симулировать сложно.

### Предлагаемый фикс (для ревью)

**Уровень 1 — application-level timeouts (быстрая защита от единичного zombie)**:
- Обернуть `write_all_flush(client_stream, &server.buffer).await?` в `src/server/protocol_io.rs:147` в `tokio::time::timeout` (например, 30s). При timeout → mark_bad + return Err → cleanup за миллисекунды, а не за `tcp_user_timeout=60s`.
- Аналогично `transaction.rs:1182` write_all_flush к client. И **`mark_bad` для НЕ-async / НЕ-copy случая** (сейчас при write fail мы тихо проглатываем error и continue loop, что усугубляет ситуацию).

**Уровень 2 — structural defense от continuous slow-client pressure (ключевое для 4-дневного залипания)**:
- Метрика "client write blocked > N секунд" — сейчас невидимо в `SHOW POOLS`/Prometheus. Добавить counter `cl_write_stalled` для observability.
- При checkout timeout `phase=semaphore` (т.е. кто-то 5 секунд ждёт permit), **forced eviction** самой старой client-task'и: kill TCP к клиенту, отозвать permit, дать новому. Это даёт self-healing и предотвращает 4-дневный outage.
- Альтернатива: per-pool **client write watchdog** — отдельный task раз в N секунд проверяет всех клиентов, у кого `last_activity > threshold`, отстреливает.

**Уровень 3 — устранить архитектурное ограничение reserve**:
- Сейчас `acquire_coordinator_jit` (где выдаётся reserve permit) вызывается **после** `acquire_semaphore`. Если pool semaphore исчерпан — reserve бесполезен.
